# Circuit Crafter — Component Generation Engine Documentation

This document provides a highly detailed, comprehensive technical specification for the Circuit Crafter component architecture. It is designed to act as the primary instruction manual for an automated Component Generation Engine capable of scraping datasheets (or PDFs) and outputting the 300+ components required for the library.

---

## 1. Architectural Philosophy: Pure Data-Driven Rendering

In Circuit Crafter v2, the React frontend contains **zero hardcoded component logic**. It operates strictly as a "dumb" interpreter. 

*   **The Engine's Job:** Output highly structured JSON files that perfectly describe the geometry, electrical properties, and visual appearance of a component.
*   **The Renderer's Job:** Read the JSON, wrap it in an `<svg>`, attach interactive `onDrag`/`onClick` handlers, and draw it on the canvas.

If a component looks wrong in the editor, it means the JSON generated by your engine is wrong.

---

## 2. The Coordinate System & Grid Mechanics

Understanding the coordinate system is critical for generating components that align perfectly with the editor's wiring system.

### 2.1 The Global vs. Local Space
*   **Global Space:** The infinite canvas where the user works. The top-left of the canvas is roughly `(0,0)`, and it scales infinitely.
*   **Local Space:** Every component defines its own local bounding box where `(0,0)` is always the top-left corner of the component's **visual body** (not including the pin lines).

### 2.2 The 20px Grid Rule
The editor enforces a strict **20px snap grid**. For components to wire together seamlessly, the engine must obey these rules:
1.  All pin `y` coordinates **must** be multiples of 20 (e.g., `20`, `40`, `60`).
2.  Component `width` and `height` should ideally be multiples of 20.
3.  The first pin on any side should generally start at `y: 20` (leaving a 20px padding from the top edge).

## 3. Comprehensive JSON Schema Specification

Every component generated by the engine must result in a single `.json` file matching this exact schema:

```typescript
{
  "id": "string",            // Unique identifier (e.g., "IC_ADC0804")
  "category": "string",      // Category for the UI picker (e.g., "ICs", "Sensors")
  "displayName": "string",   // Human-readable name (e.g., "ADC0804")
  "dimensions": {
    "width": "number",       // Width of the component body in pixels
    "height": "number"       // Height of the component body in pixels
  },
  "visuals": {
    "bodyType": "rect" | "path", 
    "pathData": "string",    // (Optional) Required ONLY if bodyType is "path". SVG path string.
    "stroke": "string",      // CSS color or "currentColor" (default: "currentColor")
    "fill": "string",        // CSS color or "none" (default: "none")
    "strokeWidth": "number"  // Thickness of the body outline (default: 2)
  },
  "defaultRotation": "number", // 0, 90, 180, or 270
  "pins": [
    {
      "id": "string",        // Unique ID for the pin within the component (e.g., "p1")
      "name": "string",      // Functional label shown to the user (e.g., "VCC", "GND", "P1.0")
      "number": "number",    // Physical pin number on the real package (1-indexed)
      "side": "left" | "right" | "top" | "bottom", // Which side of the body the pin extends from
      "position": {
        "x": "number",       // Local X coordinate WHERE THE PIN TOUCHES THE BODY
        "y": "number"        // Local Y coordinate WHERE THE PIN TOUCHES THE BODY
      },
      "electricalType": "input" | "output" | "bidirectional" | "power" | "ground" | "passive",
      "length": "number",    // (Optional) How far the pin line extends outward (default: 15)
      "invert": "boolean"    // (Optional) If true, draws a logical NOT circle at the body interface
    }
    // ... more pins
  ]
}
```

## 4. Automation Logic: Generating Standard ICs (DIP/SOIC/QFP)

For 90% of the components in your reference PDF (Microcontrollers, Logic Gates, Timers, Memory ICs), the engine should use the **Standard IC Generator Algorithm**:

### Step 1: Parse the Input Data
Assume your engine ingests a CSV/PDF table containing: `Pin Number`, `Pin Name`, `Side`.

### Step 2: Calculate Dimensions
```javascript
const PADDING = 20;
const PIN_SPACING = 20;

// Find the side with the most pins
const leftPinsCount = pins.filter(p => p.side === 'left').length;
const rightPinsCount = pins.filter(p => p.side === 'right').length;
const maxPins = Math.max(leftPinsCount, rightPinsCount);

// Calculate Height
const height = (maxPins * PIN_SPACING) + PADDING; 

// Calculate Width (Standardized based on complexity)
// Simple ICs (8-16 pins) = 80px width
// Complex MCUs (40+ pins) = 160px width
const width = maxPins > 10 ? 160 : 80;
```

### Step 3: Assign Pin Coordinates
The `position.x` and `position.y` represent where the pin *touches the body*. The React renderer automatically calculates where the wire connects based on the pin's `side` and `length`.

*   **Left Pins:**
    *   `x` is always `0`.
    *   `y` starts at `20` and increments by `20` for each subsequent pin.
*   **Right Pins:**
    *   `x` is always equal to the calculated `width` (e.g., `80` or `160`).
    *   `y` starts at `20` and increments by `20`.

### Step 4: Handle Pin Inversion (Logic NOT)
The engine must detect active-low signals.
*   **Rule:** If the `name` from the PDF contains a slash (e.g., `CS/`, `/RESET`), a bar over it, or ends in `_N` / `*`, the engine must set `"invert": true` in the JSON.
*   **Result:** The renderer will automatically strip the slash from the label and draw an inversion circle (bubble) at the pin root.

## 5. Automation Logic: Generating Custom Symbols (Resistors, Diodes, Transistors)

For discrete components and analog symbols, a rectangular body is insufficient. The engine must generate `bodyType: "path"`.

### 5.1 The `pathData` Specification
The `pathData` uses standard SVG path commands (`M` move, `L` line, `C` curve, `A` arc, `Z` close).

**Crucial Rules for Custom Paths:**
1.  **Coordinate Mapping:** The path must be drawn within the component's `dimensions` bounding box `(0,0)` to `(width, height)`.
2.  **Pin Alignment:** The `position` of the pins must perfectly align with the endpoints of the `pathData`. 

### Example: Generating a Resistor
If the engine is generating a Zig-Zag resistor:
1.  `dimensions`: `{ width: 60, height: 20 }`
2.  `pathData`: `"M 0,10 L 10,10 L 15,0 L 25,20 L 35,0 L 45,20 L 50,10 L 60,10"`
3.  **Pin Placement:**
    *   Pin 1 (Left): Must be placed exactly at the start of the path: `{ x: 0, y: 10 }`.
    *   Pin 2 (Right): Must be placed exactly at the end of the path: `{ x: 60, y: 10 }`.

### 5.2 How the Renderer interprets Pins on Custom Symbols
Unlike ICs (where `length=15` draws a line outward), for custom symbols like Resistors where the "legs" are part of the SVG `pathData`, the engine should set:
*   `"length": 0` (or let the React Pin component know it's a passive boundary).
*   In the current React implementation, the `Pin.tsx` will draw a line outward by default. Therefore, the `pathData` should only contain the *symbol itself* (the zig-zag), and the renderer will add the straight legs automatically based on the pin `length`.

## 6. Edge Cases & Special Handlers

### 6.1 Top and Bottom Pins
Some components (like Op-Amps or specific ICs) require pins on the top or bottom.
*   **Top Pins:** `position.y` is `0`. `position.x` is determined by spacing. `side` must be `"top"`. The renderer will draw the pin pointing upwards.
*   **Bottom Pins:** `position.y` is equal to `height`. `side` must be `"bottom"`. The renderer will draw the pin pointing downwards.

### 6.2 Multi-part Components (e.g., Quad Op-Amps)
Currently, Circuit Crafter treats every placed component as a single visual entity. For an IC like the LM324 (Quad Op-Amp):
*   **Approach A (Single IC Block):** Generate it as a standard 14-pin rectangle.
*   **Approach B (Discrete Gates):** Generate a JSON file for `OPAMP_GENERIC` with 5 pins (In+, In-, Out, V+, V-). The user places 4 of them. The engine does *not* need to handle heterogeneous multi-part logic internally; it just generates the available symbols.

## 7. Validating the Engine's Output

Before injecting a generated JSON into the `src/library/` folder, the generation script should run a validation pass:

1.  **Unique IDs:** Are all component `id`s globally unique?
2.  **Pin ID Uniqueness:** Are all pin `id`s unique *within* the component?
3.  **Grid Compliance:** Are all `pin.position.y` values integers and multiples of 20?
4.  **Bounds Checking:** Do all `pin.position.x` and `y` values fall within or exactly on the boundaries of `dimensions.width` and `dimensions.height`?
5.  **Side Geometry Match:** 
    *   If `side == 'left'`, `x` must be `0`.
    *   If `side == 'right'`, `x` must be `dimensions.width`.
    *   If `side == 'top'`, `y` must be `0`.
    *   If `side == 'bottom'`, `y` must be `dimensions.height`.

If your automation engine follows this document strictly, the React frontend will flawlessly render, snap, wire, and simulate any of the 300+ components instantly without frontend code changes.
